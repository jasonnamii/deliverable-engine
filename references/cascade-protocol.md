# Cascade Protocol — 타 스킬 후속 발동 상세

산출물 엔진이 타 스킬 뒤에서 자동 적용될 때의 구체적 절차와 변환 규칙.

---

## 발동 판정

| 조건 | 발동 | 이유 |
|------|------|------|
| 타 스킬이 .md 산출물 파일을 생성 | **자동 적용** | 산출물=구조화된 서술 |
| 타 스킬이 대화 내 인라인 응답만 | 미적용 | 파일 미생성 |
| 타 스킬이 .xlsx/.pptx 등 비md | 미적용 | 해당 포맷 스킬 관할 |
| 타 스킬 산출물 | **3티어 게이트** | §0 용도×길이 매트릭스 참조 |

**"자동 적용"의 의미:** 타 스킬의 분석·진단이 완료된 후, 그 결과를 .md 파일로 출력하는 시점에서 산출물 엔진의 모래시계(§2) + 블록설계(§3) + 제목규칙(§4)을 적용한다. 타 스킬의 분석 로직을 변경하는 것이 아니라, **출력 형태를 변환**하는 것이다. 적용 범위는 §0 3티어 게이트(용도×길이)에 따라 풀/경량/패스스루로 결정.

## 적용 순서 (5단계 — ⓑⓒ 병합)

```
타 스킬 분석 완료 → ⓐ spine 추출 → ⓑ 모래시계+블록설계 통합 작성 → ⓒ 제목 변환 → ⓓ QC → ⓔ 디자인
```

| 단계 | 할 일 | 구체적으로 |
|------|-------|-----------|
| ⓐ spine 추출 | 분석 결과에서 핵심 판정 1문장 도출. **실패 시 → 참조형 폴백** (아래 참조) | "이 문서의 결론은 무엇인가" — 그것이 spine |
| ⓑ 모래시계+블록설계 통합 | spine 배치 + 밀도교대 + 전환장치를 **1회 작성에서 동시 적용** | 2패스 재작성 ✗. 단일 지시: "모래시계 구조로 배치하면서 블록 밀도를 교대 배치하라" |
| ⓒ 제목 변환 | 라벨형→주장형 (설득형 문서시) | "패턴 대조"→"6개 축 전부 실패 패턴과 매칭된다" |
| ⓓ QC | **티어별 범위** — 풀: 4축 전수 / 경량: ①숫자+③VOCAB만 / 패스스루: 스킵 | 산출물 엔진 §1-③ 참조 |
| ⓔ 디자인 | design-skill cascade | 풀 적용시 필수. 내부/개인 문서는 형 요청시만 |

### spine 추출 실패 폴백 — 참조형 자동 분류

spine 추출 시도 시 아래 조건에 해당하면 **설득형이 아닌 참조형 문서로 자동 분류**:

| 실패 유형 | 예시 | 처리 |
|----------|------|------|
| 순수 데이터 | 전체가 표·수치·리스트로만 구성 | 모래시계 면제, 블록설계+QC만 적용 |
| 리스트 온리 | 체크리스트, 항목 나열 | 라벨형 제목 허용, 1블록1목적+여백구분만 |
| 다중 동등 결론 | 3개 이상의 병렬 판정, 하나로 수렴 불가 | 매트릭스/비교표로 재구성, 모래시계 면제 |

**참조형 폴백 적용 시:** ⓑ에서 모래시계 구조를 강제하지 않음. 블록설계(밀도교대·전환장치)와 QC는 티어에 따라 적용. 제목은 라벨형 허용.

## 밀도 경계 판정

밀도 ENUM(고/중/저) 분류 시 경계 사례의 승격 규칙:

| 밀도 | 기본 구성 | 경계 승격 규칙 |
|------|----------|---------------|
| 고 | 데이터·수치·비교표 | — |
| 중 | 논증·분석 | 수치 3개↑ 포함 → 고로 승격. 직접인용 2개↑ → 중 유지(저로 하락 ✗) |
| 저 | 서사·맥락·전환 | 인용 1개 포함 → 저 유지. 분석적 서사(인과 추론 포함) → 중으로 승격 |

## 축별나열 → 블록전환 변환

타 스킬(biz-skill, risk-radar, research-frame 등)은 본질적으로 "축별 분석"을 한다. 이 결과를 그대로 출력하면 **동일 밀도 N연속 나열** — 블록설계 규칙6(밀도대비) 위반이 된다.

### 안티패턴: "사전식 나열"

```
## 패턴 대조                     ← 라벨형 제목
자본 축에서는 A다. (중밀도 논증)
플랫폼 축에서는 B다. (중밀도 논증)
규제 축에서는 C다. (중밀도 논증)
제휴 축에서는 D다. (중밀도 논증)
시장 축에서는 E다. (중밀도 논증)
→ 5연속 동일 밀도 = 규칙6 위반
→ 축 라벨 전환만 = 규칙4 위반
```

### 변환 원칙

| # | 원칙 | 방법 |
|---|------|------|
| 1 | **축을 서사로 엮어라** | 축별 나열 ✗ → 인과·시간·논증 흐름으로 재배치. "자본이 없으면→플랫폼 불가→시장 신뢰 붕괴"처럼 축간 영향 맵이 서사의 뼈대가 된다 |
| 2 | **밀도를 교대하라** | 서사(저밀도) → 데이터/표(고밀도) → 논증(중밀도) 교대. 3축 분석이면 "서사도입→표→논증→서사전환→표→논증" |
| 3 | **부록으로 분리하라** | 전체 축 스크리닝 결과표, 매칭 근거표 등 참조용 데이터는 부록으로. 본문에는 핵심 축만 서사로 |
| 4 | **블록 전환 장치를 삽입하라** | 축과 축 사이에 "원인→결과", "전제→확장", "긴장→해소" 중 하나를 강제. "~축에서도" "~측면에서 보면" 같은 라벨 전환 금지 |

### Before/After 예시

**Before (축별 나열):**
```markdown
## 패턴 대조 서사
자본 구조가 부재하다. Zoom은... WeWork는... (6문단)
플랫폼 측면에서... Airbnb는... Quibi는... (6문단)
규제 축에서도... Google은... SEC는... (6문단)
```

**After (서사 엮기 + 밀도 교대):**
```markdown
## 6개 축이 하나의 실패 패턴을 가리킨다

이 문서에는 단위경제학이 없다. [결론 먼저 — 고밀도 1문장]

Zoom이 $1.8M 시드에서 $100B IPO까지 도달한 변수는 CAC payback 집착이었다.
이 문서는 그 개념 자체가 부재하다. [서사 — 저밀도]

| 축 | 판정 | 매칭 패턴 |
|...|...|...|  [표 — 고밀도, 핵심 3축만]

자본이 없으면 플랫폼을 만들 수 없고, 플랫폼이 없으면 시장 신뢰가 붕괴한다.
세 축이 상호 강화하는 하향 나선이다. [인과 엮기 — 중밀도]
```

## 스킬별 cascade 주의사항

| 상위 스킬 | 산출물 특성 | cascade 시 주의 |
|-----------|------------|----------------|
| biz-skill | 18축 스크리닝 + 패턴 매칭 | 18축 표→부록, 본문은 핵심 축만 서사로 |
| risk-radar | 확률×임팩트 매트릭스 | 매트릭스→부록 또는 본문 내 고밀도 블록, 서사로 감싸기 |
| research-frame | 축별 리서치 | 축별 파일은 그대로, 통합 산출물 생성 시 cascade 적용 |
| planning-skill | 6Phase 기획 | Phase별 나열→인과/시간 서사로 재구성 |
| person-profiler | 7축 분석 | 7축 전수 나열 ✗ → DNA 5~7대를 서사 spine으로 |

## Gotchas

| 함정 | 대응 |
|------|------|
| 타 스킬 분석 로직을 변경 | cascade는 출력 형태만 변환. 분석 자체는 타 스킬 관할 |
| spine 없이 블록설계만 적용 | spine 추출이 선행. 실패 시 참조형 폴백 (위 참조) |
| 부록 제거 | 부록은 삭제 ✗. 본문에서 빼되 문서 내 보존 |
| 3티어 게이트 무시하고 일률 풀 적용 | §0 용도×길이 매트릭스 확인 필수. 내부 20행↓ = 패스스루 |
| ⓑⓒ를 2패스 재작성 | 통합 단계에서 1회 작성으로 동시 적용. 2패스 ✗ |
| spine 강제로 데이터 문서 왜곡 | 순수 데이터/리스트/다중 동등 결론 → 참조형 폴백 |
| QC를 일률 4축 전수 | 티어별 QC 범위 참조. 경량=①숫자+③VOCAB만, 패스스루=스킵 |
